# Sandbox: verify without a real domain

Any domain ending in `.test` is a sandbox domain. Claiming one never touches real DNS — it's routed to an in-memory fixture that scripts a specific verification journey on demand, so you can experience every state (verified, pending, wrong record, unreachable, expired) without owning a domain or waiting on real propagation.

`.test` is reserved by [RFC 6761](https://www.rfc-editor.org/rfc/rfc6761) for documentation and testing — it can never be delegated in real DNS, so a sandbox domain can never collide with a real owner's domain. It's a safe, permanent namespace, not a convention that happens to work today.

Sandbox domains only work with a test-mode API key (`dp_test_...`). Claiming one with a live-mode key returns `400 sandbox_requires_test_mode`.

## The journeys

The first label of a sandbox domain (before any `.test` suffix) selects which journey it scripts. Every journey is deterministic and timed from when the domain was claimed — nothing here depends on wall-clock time beyond that.

| Domain | What happens |
| --- | --- |
| `verified.test` | The correct record is present from the very first check. Verifies immediately — models an owner who published the record before starting verification. |
| `pending-then-verified.test` | Reports `not_found` (nxdomain) until 45 seconds after the claim, then the correct record appears. Models real DNS propagation delay: long enough to feel real on a polling UI, short enough to still fit inside a live demo. |
| `wrong-value.test` | Always returns a syntactically valid but incorrect record. Every check reports `wrong_value` with a detected value that never matches the expected one — useful for exercising the expected-vs-detected diff. |
| `nxdomain.test` | Never resolves. Every check reports `not_found`, forever — models an owner who never added the record. |
| `flaky.test` | Alternates every 30 seconds between unreachable (a simulated timeout) and the correct record. Drives the `temporarily_failed` grace-window story and its recovery UX. |
| `conflict.test` | Resolves the correct record immediately, exactly like `verified.test`. DNS has no notion of "this domain is already claimed by someone else" — that's enforced by the claim endpoint itself (`409 domain_already_claimed`), not simulated in DNS. Claim `conflict.test` twice under the same mode to see it. |

## Parallel runs

Append `+<anything>` to a journey's label to get an independent domain that scripts the same journey without colliding with another claim — for example `verified+run1.test` and `verified+run2.test` behave identically but are separate claims. This is what lets a test suite (or two people demoing at once) each get their own sandbox domain per journey.

## Trying it

Claim any of these the same way you'd claim a real domain — see the [Quickstart](/docs/quickstart) for the full request/response shapes, and [Claim and verify a domain](/docs/claim-and-verify) for the complete integration guide.

```bash
curl https://api.domainproof.dev/v1/domains \
  -H "Authorization: Bearer dp_test_k7h2m5p3qrtx_a3b4c5d6e7f2g3h4i5j6k7l2m3" \
  -H "Content-Type: application/json" \
  -d '{"domain": "flaky.test"}'
```

Then call `POST /v1/domains/:id/verify` a few times, roughly 30 seconds apart, and watch the outcome alternate between `unreachable` and `found`.
