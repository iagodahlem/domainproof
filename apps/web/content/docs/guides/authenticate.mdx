# Authenticate with an API key

Every request to the Public API carries an API key in the `Authorization` header:

```bash
curl https://api.domainproof.dev/v1/domains \
  -H "Authorization: Bearer dp_test_k7h2m5p3qrtx_a3b4c5d6e7f2g3h4i5j6k7l2m3"
```

A key always looks like `dp_<mode>_<key id>_<secret>`. The `key id` segment is public — safe to show in logs or dashboards — and exists purely to look the key up; the trailing `secret` is what actually authenticates the request. An invalid, unknown, or revoked key all return the same `401 invalid_api_key` — there's no way to distinguish "this key doesn't exist" from "this key exists but the secret is wrong."

## Test vs. live

Every key is minted for one of two modes:

- **`dp_test_...`** — test-mode keys. Anything claimed with one lives entirely separate from your live data, and only test-mode keys can claim [sandbox domains](/docs/sandbox).
- **`dp_live_...`** — live-mode keys, for real domains and real DNS.

A project's test and live domains, keys, and events never mix — a test-mode key can't see or touch anything claimed under a live-mode key, and vice versa.

The rule that trips people up: **sandbox (`.test`) domains can only be claimed with a test-mode key.** Trying to claim `verified.test` (or any other sandbox domain) with a `dp_live_...` key returns:

```json
{ "error": { "code": "sandbox_requires_test_mode", "message": "Sandbox domains are only available with test keys." } }
```

## The three auth planes, at a glance

DomainProof's API is split into three authentication planes by path prefix — everything on this page and the rest of the guides is about the Public API, the one you'll integrate against directly:

| Plane | Path prefix | Authenticated by | Who calls it |
| --- | --- | --- | --- |
| Public API | `/v1/*` | Project API key (this page) | Your backend, directly — this is what the rest of these guides cover. |
| Frontend API | `/frontend/*` | A per-claim `frontendToken`, no key at all | The hosted verification page, and drop-in components — never your backend's own API key. |
| Dashboard API | `/dashboard/*` | A signed-in session | The DomainProof dashboard itself — not meant to be called directly by an integration. |

See the [API reference](/docs/api-reference) for the full plane/endpoint breakdown.
